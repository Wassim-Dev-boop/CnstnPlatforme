🏛️ Architecture Globale & Séparation des Couches

Vue d'ensemble : la plateforme suit une architecture microservices en 6 couches logiques. Chaque couche a une responsabilité unique (Single Responsibility Principle) et communique avec les couches adjacentes par des protocoles standardisés (HTTPS/REST, JDBC, Kafka). La sécurité est appliquée à chaque couche : authentification à l'entrée (Keycloak), autorisation au service (@PreAuthorize), chiffrement en transit (TLS).

2Rôle de chaque couche

1 Couche Présentation — Le navigateur & l'application Angular Angular 21 · TypeScript · Signals

C'est l'interface homme-machine (IHM). Elle se charge de : (1) afficher les écrans (dashboards, formulaires, GED, listes), (2) capturer les actions utilisateur, (3) appeler les APIs backend via HTTP. Aucune logique métier côté frontend — uniquement de la mise en forme, validation de surface et orchestration d'appels. L'authentification se fait via Keycloak (PKCE flow), le JWT est stocké en mémoire (volatile) et joint à chaque requête sortante via un intercepteur HTTP.

  • 📂 frontend/src/app/core/services/ — appels API typés
  • 📂 frontend/src/app/modules/ — modules métier (ged, events, reservations...)
  • 📂 frontend/src/app/shared/ — composants UI réutilisables (boutons, modals, toasts)
2 Couche Edge — API Gateway & Keycloak (point d'entrée unique) Spring Cloud Gateway · Keycloak 24

Point d'entrée unique pour tous les clients (port 8080). Le gateway fait 3 choses critiques : (1) Router les requêtes vers le bon microservice selon le path (/api/v1/events/** → event-service), (2) Authentifier en validant le JWT Keycloak et en extrayant les rôles, (3) Filtrer les requêtes non autorisées (CORS, rate limiting, headers de sécurité). Keycloak gère toute l'IAM (Identity & Access Management) : base d'utilisateurs, mots de passe, OAuth2, OpenID Connect.

  • 🔐 backend/api-gateway/ — Spring Cloud Gateway + JwtAuthConverter
  • 🔐 backend/keycloak/ — realm-export.json, clients configurés, users seedés
3 Couche Control Plane — Discovery & Config centralisés Eureka · Spring Cloud Config

Deux services transverses d'infrastructure as code. Eureka est l'annuaire dynamique où chaque microservice s'enregistre au démarrage (nom + URL). Le gateway demande à Eureka où est event-service plutôt que d'avoir une IP en dur → scaling horizontal transparent. Config Server centralise les application.yml de tous les services (DB, Keycloak, Eureka URL) — un seul endroit à modifier pour changer la config globale. Les services récupèrent leur config au boot via spring.config.import.

  • 🧭 backend/eureka-server/ — annuaire de services (port 8761)
  • ⚙️ backend/config-server/ — repo Git des configs (port 8888)
4 Couche Métier — 8 microservices autonomes (bounded contexts DDD) Spring Boot 3.2.5 · Java 21 · JPA · Flyway

Le cœur du système. Chaque microservice est autonome et applique le pattern Controller → Service → Repository :

  • Controller : reçoit les requêtes HTTP, valide les DTOs, vérifie les permissions via @PreAuthorize, renvoie une réponse JSON.
  • Service : contient toute la logique métier (workflow, calculs, validations croisées), orchestre les repositories et les appels vers d'autres services (Feign client).
  • Repository : interface Spring Data JPA qui abstrait l'accès SQL. Les requêtes complexes utilisent @Query ou les Specification JPA.
  • Entity : modèle du domaine mappé sur une table PostgreSQL. Mappers manuels (Entity ↔ DTO) pour découpler la persistance de l'API publique.
  • Migration : Flyway V1__*.sql à V23__*.sql pour 4 services, JPA ddl-auto pour 2 services. Idempotent, traçables.
5 Couche Persistance — PostgreSQL par service + Messaging PostgreSQL 16 · Kafka 3.7 · MailHog

Chaque microservice possède sa propre base de données (pattern DB-per-service) : aucun microservice n'accède directement à la base d'un autre. Pour les références croisées (ex : un événement référence un utilisateur), on passe par des appels REST ou Feign. Cela garantit l'isolation des données et la liberté technologique. Kafka sert de bus d'événements asynchrone pour les notifications inter-services. MailHog capture tous les emails sortants en développement (interface web localhost:8025).

  • 8 bases : auth_db, event_db, reservation_db, ged_db, intervention_db, notification_db + 2 partagées (keycloak, eureka)
  • Topic Kafka : notifications.events (inter-service pub/sub)
  • MailHog : SMTP capturé sur :1025, UI sur :8025
6 Couche Transverse — Préoccupations partagées par toutes les couches Spring Security · SLF4J · Docker · Flyway

Ces préoccupations s'appliquent horizontalement à toutes les couches : Sécurité (JWT validé à l'entrée, @PreAuthorize dans chaque service, CORS centralisé au gateway), Observabilité (logs structurés SLF4J par service, prêt pour Prometheus/Grafana), Migration de schéma (Flyway pour 4 services, JPA pour 2, versionnée et idempotente), Containerisation (Docker multi-stage, image JRE-only, orchestration Docker Compose).

  • 🔒 @PreAuthorize("hasRole('...')") dans CHAQUE méthode sensible
  • 📋 Logs SLF4J avec contexte (requestId, userId) pour traçabilité
  • 🐳 Dockerfile.service multi-stage Maven → image finale eclipse-temurin:21-jre (~250 MB)

3Flux d'une requête type (de bout-en-bout)

Scénario : L'employé "Wassim" clique sur "Publier un document" dans la GED.
  1. 🖱️ Angular (1) envoie PUT /api/v1/ged/documents/{id}/publish avec le JWT dans le header.
  2. 🛡️ API Gateway (2) valide le JWT, extrait role=ADMIN, route vers ged-service:8082.
  3. 🧭 Eureka (3) fournit l'URL réelle du pod ged-service au gateway.
  4. 🧩 ged-service (4) reçoit la requête. @PreAuthorize("hasAuthority('PUBLISH_DOCUMENT')") vérifie les permissions.
  5. 📋 DocumentService charge le doc depuis ged_db, change le statut à PUBLISHED, enregistre.
  6. 📨 ged-service publie un événement sur Kafka (5) topic notifications.events.
  7. 🔔 notification-service consomme l'événement, crée la notif, l'envoie via SSE au frontend.
  8. 📧 EmailService (MailHog en dev) envoie un email si la priorité ≥ HIGH.
  9. ✅ Réponse 200 OK remonte : Gateway → Angular. UI met à jour le statut dans la liste.

Q1 Pourquoi une architecture microservices plutôt qu'un monolithe ? Architecture
💬 Réponse

Le projet vise 8 domaines métier distincts (auth, events, reservations, GED, IT, interventions, notifications, workflow admin) avec des rythmes d'évolution différents. Le monolithe aurait obligé à redéployer tout pour un changement local. Les microservices permettent : (1) scalabilité indépendante (la GED a plus de charge que l'admin workflow), (2) isolation des pannes (un crash du service notification ne paralyse pas les events), (3) équipes découplées, (4) polyglotte (chaque service peut migrer vers un autre stack si besoin). Inconvénients assumés : complexité opérationnelle (8 déploiements, gateway, discovery) justifiée par la taille du projet.

🛠 Preuve

Q2 Comment garantissez-vous la sécurité bout-en-bout ? Sécurité
💬 Réponse

Triple niveau de défense : (1) SSO Keycloak délivre un JWT signé RS256 — l'utilisateur s'authentifie une fois, l'API Gateway valide le token sur chaque requête via RealmRoleConverter, (2) Spring Security au niveau méthode : @PreAuthorize("hasRole('ADMIN')") dans chaque controller, (3) permissions atomiques (custom authority) pour les cas granulaires : @PreAuthorize("hasAuthority('PUBLISH_DOCUMENT')"). La communication est chiffrée en HTTPS (TLS 1.3) en production. Les mots de passe ne sont jamais stockés en clair — Keycloak les hash en bcrypt.

Q4 Que se passe-t-il si Keycloak tombe ? Résilience
💬 Réponse

C'est un SPOF assumé. En production, on déploie Keycloak en cluster (2-3 instances derrière un load balancer, base PostgreSQL répliquée). Si Keycloak est totalement down, aucun nouvel utilisateur ne peut se connecter, mais les sessions en cours (JWT déjà émis) restent valides jusqu'à expiration du token (15 min par défaut). Les microservices continuent de fonctionner en mode dégradé pour les appels internes. C'est le compromis classique : centraliser l'auth simplifie la sécurité mais crée ce SPOF.

Q7 Pourquoi ne pas avoir utilisé MongoDB ou un stockage objet pour la GED ? GED

Q8 Comment évitez-vous les conflits de version dans la GED ? GED
💬 Réponse

Pas de merge complexe type Git. Modèle simple : append-only. Chaque ajout de version crée une nouvelle ligne dans document_versions avec un version_number incrémenté automatiquement. Le document parent pointe vers la version courante. Si deux utilisateurs téléversent en même temps, le premier gagne (verrou optimiste via @Version JPA), le second reçoit un 409 Conflict et doit rafraîchir. Pour le cas d'usage GED (procédures, formulaires), c'est suffisant — pas d'édition collaborative temps réel.

Q9 Pourquoi SSE plutôt que WebSocket pour les notifications ? Notif
💬 Réponse

SSE (Server-Sent Events) est unidirectionnel serveur→client, parfait pour des notifications (le client ne remonte pas de données temps réel). Avantages sur WebSocket : (1) HTTP standard — passe les proxies, load balancers, CDN sans config, (2) reconnexion automatique native du navigateur, (3) Spring Boot l'implémente nativement via SseEmitter, (4) plus simple à débugger (c'est du HTTP). WebSocket serait utile pour un chat bidirectionnel, pas pour des notifs.

Q11 Pourquoi Angular 21 et pas React ou Vue ? Frontend
💬 Réponse

Angular est un framework opinionated (structure imposée, DI, RxJS, CLI) — idéal pour un projet académique où la cohérence d'équipe est critique. React nécessite des choix supplémentaires (Redux vs Zustand, react-router vs Next, etc.). Angular fournit tout en boîte. Angular 21 apporte les Signals (state management fin et réactif sans RxJS obligatoire), les standalone components (plus de NgModule), et une meilleure DX. Pour un intranet d'entreprise, Angular est un choix conservateur et professionnel.

Q14 Pourquoi avoir choisi Spring Cloud plutôt que Kubernetes natif pour le service discovery ? Architecture
💬 Réponse

Eureka est plus simple à mettre en place et à débugger pour un projet académique. K8s natif (DNS, headless services) est plus performant mais nécessite tout un cluster K8s. Pour ce projet Docker Compose, Eureka fait le job sans overhead. Migration possible : supprimer Eureka, garder les noms de services, basculer sur K8s DNS. C'est l'avantage d'un standard ouvert.

Q15 Comment avez-vous géré la réinitialisation de mot de passe sans révéler l'existence du compte ? Sécurité
💬 Réponse

L'endpoint POST /forgot retourne toujours 200 OK avec un message générique, que l'email existe ou non. Le système envoie le mail de reset uniquement si le compte existe. Empêche l'énumération (un attaquant ne peut pas deviner quels emails sont enregistrés). Le token de reset est : (1) hashé en DB (pas de fuite en cas de dump), (2) à usage unique (marqué consommé), (3) à durée de vie courte (15 min), (4) lié à l'IP et au user-agent pour détecter les anomalies.

Q16 Quels sont les risques d'un JWT et comment les mitigez-vous ? Sécurité
💬 Réponse

Risques : (1) vol de token → mitigé par HTTPS, durée courte (15 min), refresh token en cookie HttpOnly, (2) révocation impossible sans state → mitigé par une blacklist Redis des tokens révoqués, (3) taille du token dans chaque requête → on garde le JWT minimal (sub, roles, exp, iat), (4) rejeu (replay attack) → mitigé par le nonce jti (JWT ID) unique. Pour une app interne au CNSTN, c'est un compromis acceptable. Pour une app publique, on préférerait des sessions serveur.

  • backend/auth-user-service/.../JwtAuthConverter.java (extraction des rôles)
Q17 Comment fonctionne la validation multi-niveaux (Manager → Sécurité → DSN) ? Event
💬 Réponse

Une machine à états finis (FSM). L'événement a un champ workflow_step parmi BROUILLAN, VALIDATION_MANAGER, VALIDATION_SECURITE, VALIDATION_DSN, VALIDATION_SALLE, TERMINE, REFUSE. Chaque transition est une méthode XDecision dans EventService qui : (1) vérifie que l'utilisateur a le bon rôle via @PreAuthorize, (2) enregistre la transition en base, (3) passe au step suivant ou rejette. Le service refuse les transitions invalides (ex : un DSN ne peut pas valider avant le manager). Toute l'historique est dans event_workflow_history.

Q18 Comment garantir que l'email de notification ne se perde pas ? Notif
💬 Réponse

Chaque tentative d'envoi est tracée dans notification_email_logs avec : notification ID, destinataire, sujet, statut (PENDING / SENT / FAILED / SKIPPED), raison d'échec, horodatage. En cas de FAILED, un endpoint POST /resend/{notificationId} permet à l'admin de relancer manuellement. Le SMTP est configuré avec retry (3 tentatives, backoff exponentiel). Un email est considéré livré quand SMTP retourne 250 OK. Aucune perte silencieuse.
